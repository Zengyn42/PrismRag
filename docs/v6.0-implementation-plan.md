# PrismRag v6.0 联邦元图：实现计划

> 日期：2026-05-26
> 状态：设计定稿，待实现
> 依赖：v5.7 完成（✅）

---

## 架构一句话总结

> **物理隔离、逻辑统一**：每个 namespace 独立存储和写入；serve-time 通过三层 bridge 构建只读合并视图；写操作增量热补丁内存图；聚类定期后台重建。

---

## 被否定的方案（禁止实现）

| 方案 | 否定原因 |
|------|---------|
| Leiden centroid 代表点 | 结构聚类中心 ≠ 语义代表，是语义噪声 |
| 纯 lazy bridge | 破坏全局图算法（Leiden、impact analysis 无骨架可走）|
| 中央单一 registry.json | 幽灵指针 + filelock 并发瓶颈 |
| 持久化 merged 模式 | 回滚成本 O(n)，写锁复杂度不可控 |
| `fcntl.flock` | 跨平台不兼容（Windows WSL 边界问题）|
| Hub 节点集存入 manifest.json | 派生数据不属于 source of truth，应存 registry_cache |
| BM25 同步更新 | rank_bm25 不支持增量，需全量重建，必须异步 |

---

## 核心设计决策

### 1. Manifest Schema（source of truth）

每个 namespace 在 `.prismrag/manifest.json` 中存储**纯源头信息**，不存派生数据：

```json
{
  "schema_version": 1,
  "namespace": "pulsify",
  "vault_path": "/home/kingy/Projects/Pulsify",
  "data_dir": "/home/kingy/Projects/Pulsify/.prismrag/pulsify",
  "ingest_time": "2026-05-26T10:30:00Z",
  "node_count": 1240,
  "edge_count": 3820,
  "embedding_model": "ollama/bge-m3",
  "embedding_dim": 1024,
  "prismrag_version": "5.7.0"
}
```

**不在 manifest 里的字段**（派生数据，存 registry_cache）：
- `hub_node_ids` — 由 degree 排序计算得来
- `hub_set_version` — Hub 节点集版本号
- `community_count` — 可从 graph.json 派生

### 2. Registry Cache Schema

`~/.prismrag/registry_cache.json`：

```json
{
  "schema_version": 1,
  "entries": {
    "pulsify": {
      "manifest_path": "/home/kingy/Projects/Pulsify/.prismrag/pulsify/manifest.json",
      "last_seen": "2026-05-26T10:30:00Z",
      "hub_node_ids": ["pulsify::AudioEngine", "pulsify::TrackManager", ...],
      "hub_set_version": 3,
      "community_count": 42
    }
  }
}
```

- Serve 启动时对每条 `manifest_path` 做 `stat` 探活（< 1ms），自动剔除幽灵条目
- 写入用 `filelock` 库 + 原子重命名（`write tmp → os.replace`）

### 3. 三层 Bridge 构建

```
Serve 启动时：
  第一层（确定性，零成本）
    → shared tag bridge：两个 namespace 有相同 #tag → bridge edge
    → import/dependency bridge：代码 import 跨 namespace → bridge edge

  第二层（有界 ANN，< 1s）
    → Hub nodes：每个 namespace 取 top-30 by degree（不用 betweenness，O(VE) 太贵）
    → 随机采样 20 个节点作为补充代表点
    → 跨 namespace 只对 50 个代表点做 embedding 比较
    → 代表点命中（余弦 > 0.70）→ 建立所在 community 间的骨架 bridge
    → Hub 节点集和版本号写入 registry_cache

Query-time（lazy，按需）：
  第三层
    → scope=all 查询 或 bridge miss → 展开节点级 ANN
    → LRU 缓存，key = (source_ns, source_hub_set_version,
                       target_ns, target_hub_set_version, query_embedding_hash)
    → 版本号不匹配 → 缓存失效，重新计算
```

**Hub 节点集变更时（后台 rebuild 完成）**：
1. `hub_set_version` + 1
2. LRU 缓存自动失效（key 含版本号）
3. 不需要手动清空缓存

### 4. 写操作热补丁

```
MCP write_note / patch_note 调用后：
  1. 写入物理 vault 文件（原有逻辑）
  2. 增量 ingest 新节点到对应 namespace 的 KnowledgeGraph（内存）
  3. 新节点标记 community_id = "inbox"（未聚类占位符）
  4. 注入内存合并视图（追加 namespace::id 节点）
  5. BM25 索引 → 异步队列（后台重建，写操作不等待）
  6. Lance 向量索引 → 异步队列（后台重建）
  7. 返回成功给 MCP client（毫秒级响应）

后台异步任务（每 30 分钟 或 --rebuild 触发）：
  - 重建 BM25 索引
  - 重建 Lance 向量索引
  - 重新 Leiden 聚类
  - 更新 hub_node_ids + hub_set_version
  - 重建骨架 bridge（第二层）
```

**一致性承诺**：新节点写入后，图遍历（BFS/DFS）立即可达；hybrid search（BM25 + ANN）有最长 30 分钟不可见窗口。这是可接受的一致性延迟，不是数据错误。

### 5. 联邦可视化：混合方案

```
默认视图（meta-graph，静态生成）：
  prism-rag visualize --federated
  → ~/.prismrag/meta_graph.html
  → 每个 namespace 一个超级节点（大小 = node_count）
  → bridge edges 显示为虚线（粗细 = bridge 数量）
  → 点击超级节点 → 打开 .prismrag/<ns>/graph.html（portal link）

单 namespace 视图（已有，不变）：
  .prismrag/<ns>/graph.html
  → 现有 force-graph，portal nodes 指向其他 namespace

跨图查询结果视图（动态生成）：
  当 MCP tool 返回跨 namespace 结果时，可选生成：
  → /tmp/prismrag_query_{timestamp}.html
  → 只包含查询相关节点 + bridge edges
  → 只读，不支持进一步探索
  → 通过 [SEND_FILE: ...] 发给 Discord
```

```python
def _select_hub_nodes(self, namespace: str, k: int = 30, m: int = 20) -> list[str]:
    """
    Select k top-degree nodes + m random nodes as bridge representatives.
    Returns node IDs (no namespace prefix).
    """
    graph = self._graphs[namespace]
    degrees = sorted(graph.g.degree(), key=lambda x: -x[1])
    hub = [n for n, _ in degrees[:k]]
    pool = [n for n, _ in degrees[k:]]
    random_sample = random.sample(pool, min(m, len(pool)))
    return hub + random_sample
```

### P2：写操作热补丁 + 后台 Rebuild

**依赖**：P1 完成

#### P2.1 — 热补丁注入

**文件**：`prism_rag/mcp_server/server.py`

```python
async def _hot_patch(namespace: str, new_node: Node):
    """Inject new node into in-memory graph immediately after write."""
    graph = _federated._graphs[namespace]
    graph.add_node(new_node)
    # Mark as unclustered
    graph.g.nodes[new_node.id]["community_id"] = "inbox"
    # Schedule async rebuild
    asyncio.create_task(_async_rebuild(namespace))
```

#### P2.2 — 异步 Rebuild 任务

```python
async def _async_rebuild(namespace: str):
    """Background: rebuild BM25, Lance, Leiden, hub nodes for namespace."""
    await asyncio.sleep(0)  # yield to event loop
    # 1. rebuild BM25
    # 2. rebuild Lance
    # 3. re-run Leiden
    # 4. reselect hub nodes, increment hub_set_version
    # 5. rebuild Layer-2 bridges for affected namespace pairs
```

#### P2.3 — 定时 Rebuild（可选，30分钟）

```python
async def _scheduled_rebuild():
    while True:
        await asyncio.sleep(1800)
        for ns in _federated.namespaces:
            if _has_pending_inbox_nodes(ns):
                await _async_rebuild(ns)
```

```python
def generate_meta_graph_html(
    federated: FederatedGraph,
    output_path: Path,
) -> None:
    """
    Generate a meta-graph HTML with one super-node per namespace.
    Super-node size = node_count. Bridge edges = dashed lines.
    Click super-node = portal link to namespace's graph.html.
    """
    nodes = []
    links = []
    for ns, graph in federated.graphs.items():
        nodes.append({
            "id": ns,
            "label": ns,
            "val": math.sqrt(graph.node_count) * 5,
            "color": _ns_color(ns),
            "kind": "namespace",
            "node_count": graph.node_count,
            "portal_href": f"{ns}/graph.html",
        })
    for bridge in federated.bridges:
        links.append({
            "source": bridge["source_ns"],
            "target": bridge["target_ns"],
            "relation": bridge["relation"],
            "_dashed": True,
        })
```
