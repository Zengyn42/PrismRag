# PrismRag v6.0 Federated Meta-Graph: Implementation Plan

> Date: 2026-05-26
> Status: Design finalized, pending implementation
> Dependency: v5.7 complete (✅)

---

## Architecture in One Sentence

> **Physically isolated, logically unified**: each namespace stores and writes independently; at serve-time a three-layer bridge builds a read-only merged view; write operations hot-patch the in-memory graph; clustering is periodically rebuilt in the background.

---

## Rejected Approaches (Must Not Implement)

| Approach | Reason for Rejection |
|------|---------|
| Leiden centroid representative nodes | Structural cluster centers ≠ semantic representatives — introduces semantic noise |
| Pure lazy bridge | Breaks global graph algorithms (Leiden, impact analysis have no skeleton to walk) |
| Central single registry.json | Ghost pointers + filelock concurrency bottleneck |
| Persistent merged mode | Rollback cost O(n), write-lock complexity unmanageable |
| `fcntl.flock` | Cross-platform incompatible (Windows/WSL boundary issues) |
| Storing hub node set in manifest.json | Derived data does not belong in source of truth — store in registry_cache |
| Synchronous BM25 updates | rank_bm25 does not support incremental updates; requires full rebuild; must be async |

---

## Core Design Decisions

### 1. Manifest Schema (Source of Truth)

Each namespace stores **pure source information** in `.prismrag/manifest.json` — no derived data:

```json
{
  "schema_version": 1,
  "namespace": "pulsify",
  "vault_path": "/home/kingy/Projects/Pulsify",
  "data_dir": "/home/kingy/Projects/Pulsify/.prismrag/pulsify",
  "ingest_time": "2026-05-26T10:30:00Z",
  "node_count": 1240,
  "edge_count": 3820,
  "embedding_model": "ollama/bge-m3",
  "embedding_dim": 1024,
  "prismrag_version": "5.7.0"
}
```

**Fields NOT in manifest** (derived data, stored in registry_cache):
- `hub_node_ids` — computed by degree ranking
- `hub_set_version` — hub node set version number
- `community_count` — derivable from graph.json

### 2. Registry Cache Schema

`~/.prismrag/registry_cache.json`:

```json
{
  "schema_version": 1,
  "entries": {
    "pulsify": {
      "manifest_path": "/home/kingy/Projects/Pulsify/.prismrag/pulsify/manifest.json",
      "last_seen": "2026-05-26T10:30:00Z",
      "hub_node_ids": ["pulsify::AudioEngine", "pulsify::TrackManager", ...],
      "hub_set_version": 3,
      "community_count": 42
    }
  }
}
```

- On serve startup, each `manifest_path` is probed with `stat` (< 1ms), ghost entries are automatically evicted
- Writes use `filelock` library + atomic rename (`write tmp → os.replace`)

### 3. Three-Layer Bridge Construction

```
At serve startup:
  Layer 1 (deterministic, zero cost)
    → shared tag bridge: two namespaces share a #tag → bridge edge
    → import/dependency bridge: code import crosses namespace → bridge edge

  Layer 2 (bounded ANN, < 1s)
    → Hub nodes: top-30 by degree per namespace (not betweenness, O(VE) too expensive)
    → 20 random nodes as supplementary representatives
    → Only compare embeddings across 50 representatives per namespace pair
    → Representative hit (cosine > 0.70) → build community-level skeleton bridge
    → Hub node set and version written to registry_cache

At query-time (lazy, on demand):
  Layer 3
    → scope=all query or bridge miss → expand to node-level ANN
    → LRU cache, key = (source_ns, source_hub_set_version,
                       target_ns, target_hub_set_version, query_embedding_hash)
    → Version mismatch → cache invalidated, recomputed
```

**When the hub node set changes (after background rebuild completes)**:
1. `hub_set_version` + 1
2. LRU cache automatically invalidated (key includes version)
3. No manual cache clearing needed

### 4. Write Operation Hot-Patching

```
After MCP write_note / patch_note:
  1. Write to physical vault file (existing logic)
  2. Incrementally ingest new node into the namespace's KnowledgeGraph (in-memory)
  3. New node marked community_id = "inbox" (unclustered placeholder)
  4. Inject into in-memory merged view (append namespace::id node)
  5. BM25 index → async queue (background rebuild, write does not wait)
  6. Lance vector index → async queue (background rebuild)
  7. Return success to MCP client (millisecond response)

Background async task (every 30 minutes or triggered by --rebuild):
  - Rebuild BM25 index
  - Rebuild Lance vector index
  - Re-run Leiden clustering
  - Update hub_node_ids + hub_set_version
  - Rebuild skeleton bridge (Layer 2)
```

**Consistency guarantee**: after a new node is written, graph traversal (BFS/DFS) reaches it immediately; hybrid search (BM25 + ANN) has a maximum 30-minute visibility lag. This is acceptable consistency delay, not a data error.

### 5. Federated Visualization: Hybrid Approach

```
Default view (meta-graph, statically generated):
  prism-rag visualize --federated
  → ~/.prismrag/meta_graph.html
  → one super-node per namespace (size = node_count)
  → bridge edges shown as dashed lines (thickness = bridge count)
  → click super-node → opens .prismrag/<ns>/graph.html (portal link)

Single-namespace view (existing, unchanged):
  .prismrag/<ns>/graph.html
  → existing force-graph, portal nodes pointing to other namespaces

Cross-graph query result view (dynamically generated):
  When an MCP tool returns cross-namespace results, optionally generate:
  → /tmp/prismrag_query_{timestamp}.html
  → only includes query-relevant nodes + bridge edges
  → read-only, no further exploration
  → sent to Discord via [SEND_FILE: ...]
```

#### P1.1 — Hub Node Selection

**File**: `prism_rag/store/federated.py`

```python
def _select_hub_nodes(self, namespace: str, k: int = 30, m: int = 20) -> list[str]:
    """
    Select k top-degree nodes + m random nodes as bridge representatives.
    Returns node IDs (no namespace prefix).
    """
    graph = self._graphs[namespace]
    degrees = sorted(graph.g.degree(), key=lambda x: -x[1])
    hub = [n for n, _ in degrees[:k]]
    pool = [n for n, _ in degrees[k:]]
    random_sample = random.sample(pool, min(m, len(pool)))
    return hub + random_sample
```

### P2: Write Operation Hot-Patching + Background Rebuild

**Dependency**: P1 complete

#### P2.1 — Hot-Patch Injection

**File**: `prism_rag/mcp_server/server.py`

```python
async def _hot_patch(namespace: str, new_node: Node):
    """Inject new node into in-memory graph immediately after write."""
    graph = _federated._graphs[namespace]
    graph.add_node(new_node)
    # Mark as unclustered
    graph.g.nodes[new_node.id]["community_id"] = "inbox"
    # Schedule async rebuild
    asyncio.create_task(_async_rebuild(namespace))
```

#### P2.2 — Async Rebuild Task

```python
async def _async_rebuild(namespace: str):
    """Background: rebuild BM25, Lance, Leiden, hub nodes for namespace."""
    await asyncio.sleep(0)  # yield to event loop
    # 1. rebuild BM25
    # 2. rebuild Lance
    # 3. re-run Leiden
    # 4. reselect hub nodes, increment hub_set_version
    # 5. rebuild Layer-2 bridges for affected namespace pairs
```

#### P2.3 — Scheduled Rebuild (Optional, Every 30 Minutes)

```python
async def _scheduled_rebuild():
    while True:
        await asyncio.sleep(1800)
        for ns in _federated.namespaces:
            if _has_pending_inbox_nodes(ns):
                await _async_rebuild(ns)
```

```python
def generate_meta_graph_html(
    federated: FederatedGraph,
    output_path: Path,
) -> None:
    """
    Generate a meta-graph HTML with one super-node per namespace.
    Super-node size = node_count. Bridge edges = dashed lines.
    Click super-node = portal link to namespace's graph.html.
    """
    nodes = []
    links = []
    for ns, graph in federated.graphs.items():
        nodes.append({
            "id": ns,
            "label": ns,
            "val": math.sqrt(graph.node_count) * 5,
            "color": _ns_color(ns),
            "kind": "namespace",
            "node_count": graph.node_count,
            "portal_href": f"{ns}/graph.html",
        })
    for bridge in federated.bridges:
        links.append({
            "source": bridge["source_ns"],
            "target": bridge["target_ns"],
            "relation": bridge["relation"],
            "_dashed": True,
        })
```
