## P1 — MCP 指令注明 Vault 路径

**问题**：Gemini CLI 新 session 时把 CWD 当默认工作空间，导致第一轮 `read_note` / `atomize_scan` 路径推断出错。

**修改**：在 `server.py` 顶部，`mcp = FastMCP(...)` 之前，从 `PrismRagSettings` 读取 vault 路径并注入 `instructions`：

```python
try:
    _startup_settings = PrismRagSettings()
    _vault_hint = f"Vault root: {_startup_settings.vault_path}. "
except Exception:
    _vault_hint = ""

mcp = FastMCP(
    "PrismRag",
    instructions=(
        "PrismRag: graph-first RAG system for Obsidian vaults and code repositories. "
        f"{_vault_hint}"
        "File paths in atomize_scan/read_note/write_note are relative to vault root. "
        "Exposes graph query tools (search, explain, path, communities, impact, drift) "
        "and vault CRUD tools (read, write, patch, move, delete, tags, links). "
        "Read prismrag://namespaces to see which namespace covers which codebase or vault."
    ),
)
```

Settings 在 server 启动时加载一次，无副作用。

**验收**：Jei 对话测试（P1 测试步骤）——新 session 第一轮调用 `read_note` 不出现路径推断错误。

## P2 — `explain_node` 支持 KNOW-ID 精确查找

**问题**：`search_knowledge("KNOW-000008")` 走向量语义路径，无法可靠命中数字 ID。Jei 没有意识到应该用 `explain_node` 做 ID 精确查找。

### 代码修改（`server.py`）

在 `explain_node` 函数体开头，`resolve_entry_points` 调用之前，加 KNOW-ID 先置检查。匹配后直接定位 `(ns, node_id)`，写入 `entries` 变量，后续 edge-building 逻辑不变：

```python
import re as _re

@mcp.tool()
def explain_node(node: str, scope: str = "") -> str:
    """...updated docstring below..."""
    fg = _ensure_federated()

    # Short-circuit for KNOW-ID exact lookup (e.g. "KNOW-000008").
    # Bypasses label scan — works regardless of display title changes (P4).
    entries = None
    if _re.match(r'^KNOW-\d{6}$', node, _re.IGNORECASE):
        kid = node.upper()
        target_ns = [scope] if scope else fg.namespaces
        for ns in target_ns:
            graph = fg.get_graph(ns)
            if graph is None:
                continue
            for nid, data in graph.g.nodes(data=True):
                if data.get("knowledge_id") == kid:
                    entries = [(ns, nid)]
                    break
            if entries:
                break

    if entries is None:
        entries = resolve_entry_points(fg, node, scope=scope or None)

    if not entries:
        return json.dumps({"error": f"Node not found: {node!r}"}, ensure_ascii=False)

    ns, node_id = entries[0]
    # ... rest of function unchanged (outgoing/incoming edges, community info)
```

`entries` 变量被先置检查填充后，下面的 edge-building 代码无需任何修改。

### 描述修改

**`explain_node` docstring** 更新首句：
> "Return all edges (incoming/outgoing), community membership, and full content for a single graph node. Pass a KNOW-NNNNNN ID (e.g. 'KNOW-000008') for guaranteed exact lookup — semantic search cannot reliably match numeric IDs. ..."

**`search_knowledge` docstring** 末尾加：
> "For exact KNOW-ID lookup (e.g. 'KNOW-000008'), use explain_node() instead — vector search cannot reliably match numeric IDs."

**验收**：
- 单元测试：直接调用 `explain_node("KNOW-000008")`，验证返回正确节点
- Jei 对话测试：Jei 被问及 KNOW-000008 节点内容时，是否正确调用 `explain_node` 而非 `search_knowledge`

## P3 — `list_knowledge_nodes` 内容摘要

**问题**：`list_knowledge_nodes` 只返回 `id/label/namespace/tokens`，Jei 无法基于内容做节点主题对比（E10 场景）。

**修改**：在 `server.py` 的 `list_knowledge_nodes` 函数，每个节点追加 `body_preview` 字段：

```python
results.append({
    "id": node_id,
    "label": data.get("label", node_id),
    "knowledge_id": data.get("knowledge_id"),
    "namespace": ns,
    "tokens": data.get("tokens", 0),
    "body_preview": data.get("content", "")[:100],  # 约 50–80 中文字
})
```

同步更新 docstring 字段说明，加入 `body_preview`。

**Token 开销估算**：15 节点 × 100 字 ≈ 1,500 token 增量，可接受。

**验收**：
- 单元测试：调用 `list_knowledge_nodes()`，验证每个节点含 `body_preview` 字段，长度 ≤ 100
- Jei 对话测试：Jei 被要求对所有 KNOW 节点做主题聚类，验证能给出有内容依据的分析（而非仅靠 label 猜测）

## P4 — KNOW 节点 label 使用 frontmatter title

**问题**：`list_knowledge_nodes` 返回的 `label` 字段值为 `"KNOW-000004"`（文件名），而非实际标题。

**修改**：`incremental.py` 中，知识节点 ingestion 时，label 优先取 frontmatter `title`：

```python
# 当前（第 109 行附近）：
label=doc.label,

# 修改为：
label=(doc.frontmatter.get("title") or doc.label) if kind == "knowledge" else doc.label,
```

Fallback 逻辑：无 `title` 字段时退回 `doc.label`（文件名），不影响现有行为。

**注意**：P4 改动影响**新 ingest** 的节点；已入图节点的 label 需下一次完整 `graph ingest` 才会更新。P2 的 KNOW-ID 先置检查通过 `knowledge_id` 字段匹配，不依赖 label，P4 不会破坏 P2。

**验收**：
- 单元测试：ingest 一个含 `title: "测试标题"` frontmatter 的 KNOW 文件，验证图中节点 label = "测试标题"

## 测试策略

**原则**：能确定性测试的确定性测试；凡涉及 LLM 工具选择或知识理解的行为，用 Jei 对话测试作为真正的验收标准。PrismRag 的最终目的是让 LLM 更好地理解知识集，所以对话测试是必要的，不是可选的。

| `test_list_knowledge_nodes_has_body_preview` | 每个节点有 `body_preview`，长度 ≤ 100 |
| `test_list_knowledge_nodes_preview_empty_ok` | 无 content 节点的 `body_preview` 为 `""` 不报错 |
| `test_incremental_label_uses_title` | ingest 含 title 的 KNOW 文件后，label = title |
| `test_incremental_label_fallback` | ingest 无 title 的 KNOW 文件后，label = 文件名 stem |

| 步骤 | 验证内容 | 类型 |
|------|---------|------|
