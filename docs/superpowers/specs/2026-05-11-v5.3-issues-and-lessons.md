## 一、实现阶段 Bug

### B1 — KNOW 节点 apply 后无 embedding
**发现时机**：Sprint C 完成后首轮 Jei 对话测试（T8 语义搜索）  
**症状**：`atomize_apply` 之后 `search_knowledge` 无法检索到新建的 KNOW 节点  
**根因**：`atomize_apply_impl` 对 KNOW 文件调用 `ingest_file(skip_embed=True)` 以提速，导致新节点从未进入 embedding 向量库  
**修复**：在 `atomize.py` 的 apply 完成后增加批量 embedding pass：对所有新建 KNOW 节点一次性计算 embedding、更新 LanceDB、移除旧 embedding_similar 边、调用 `link_similar_nodes` 重建相似度边、重新 Leiden 聚类并 `graph.save()`  
**Commit**：`2cb4c0a`

---

### B2 — `_maybe_reload()` 未接入读路径
**发现时机**：Sprint C 后 Jei 对话测试 T7（`list_knowledge_nodes` 返回旧数据）  
**症状**：`atomize_apply` 之后 `list_knowledge_nodes` 仍返回 apply 前的节点列表  
**根因**：`FederatedGraph` 的 `_maybe_reload()` 逻辑（500ms mtime debounce）虽然已实现，但 `get_graph()` 从未调用它——Sprint 2 的 TODO 没有接线  
**修复**：在 `federated.py` 的 `get_graph()` 开头加一行 `self._maybe_reload()`  
**Commit**：`63bf40e`

---

### B3 — `_resolve_vault()` 多命名空间报错
**发现时机**：Jei 对话测试 E1（`read_note` 失败）  
**症状**：nimbus + code 双命名空间下，调用 `read_note` 不带 namespace 参数时报错 "Multiple namespaces loaded. Specify namespace."，导致 Jei 回退到内置文件工具  
**根因**：`_resolve_vault()` 的多命名空间分支没有默认值逻辑，一律报错  
**修复**：`vault_tools.py` 中补充"取第一个非 code 命名空间"作为默认值  
**Commit**：`1b71270`

---

### B4 — `atomize_apply` 后 `_federated` 缓存不刷新（最重要）
**发现时机**：E7-E10 补测 + 对 live MCP 服务器的直接查询  
**症状**：MCP 服务器长期运行时，`atomize_apply` 写入 `graph.json` 后，`list_knowledge_nodes` 仍返回旧的节点数。重启服务器后数据正确  
**根因**：`atomize_apply` 在 `server.py` 中直接调用 `atomize_apply_impl()`，调用后没有执行 `_federated = None`。而所有 vault 写工具都通过 `_sync_graph()` 触发缓存失效，`atomize_apply` 绕过了这一机制。`_maybe_reload()` 的 mtime 检测在中间状态下已更新了缓存时间戳，后续不再重载  
**修复**：在 `server.py` 的 `atomize_apply()` 成功返回前加 `global _federated; _federated = None`  
**Commit**：`165ddcb`

## 二、测试阶段发现的行为问题（非 Bug，设计层面）

### P1 — 新 session 首轮 Gemini CWD 与 vault 不匹配
**症状**：每次新开 Jei session，E1 类型的第一轮调用 `read_note` 时 Gemini 报告 vault 路径不匹配（Gemini CLI 的 CWD = `/home/kingy/Foundation/PrismRag/`，vault 在 NimbusVault）。续 session 后恢复正常  
**性质**：Gemini CLI 将 CWD 当作默认工作空间；MCP 工具的 vault 路径在 system prompt 中未显式声明  
**建议**：v5.4 在 MCP server instructions 或工具描述中注明 vault 绝对路径，减少 LLM 的路径推断

### P2 — `search_knowledge` 不支持 KNOW-ID 精确查找
**症状**：E8 搜索 "KNOW-000008" 时，语义搜索无法命中——ID 字符串本身不携带语义信息  
**根因**：`search_knowledge` 走的是向量相似度搜索，没有 ID 精确匹配路径  
**建议**：v5.4 增加 `get_node(id: str)` 工具，支持 ID 精确查询，与 `search_knowledge` 互补

### P3 — `list_knowledge_nodes` 只返回元数据，无内容摘要
**症状**：E10 要求对比所有 KNOW 节点的主题做聚类分析，但工具只返回 `id/label/namespace/tokens`，无正文内容  
**根因**：设计时为了减少 token 消耗，省略了 body 字段  
**建议**：v5.4 增加可选参数 `include_summary=True`，返回前 200 字的内容摘要

### P4 — KNOW 节点 label 存储为 ID 而非标题
**症状**：`list_knowledge_nodes` 返回的 label 字段值为 "KNOW-000004"（即 ID 本身），而非知识节点的实际标题（如"PrismRag 与 GitNexus 的核心定位差异"）  
**根因**：`incremental.py` 用 `doc.label`（= 文件名去掉扩展名）作为 label，KNOW 文件名即为 ID  
**建议**：v5.4 在 `ingest_file` 中检查 `knowledge_id` 存在时，改用 frontmatter 的 `title` 字段作为 label

### P5 — 测试脚本 keyword check 逻辑过于脆弱
**症状**：两处测试误判：E2 检查 "proposal_id"（下划线）而 Jei 回复 "Proposal ID"（空格）；S2 检查 "成功/success/applied" 而 Jei 回复 "已完成/已存入"  
**建议**：关键词匹配改为语义检查，或检查更广义的中文完成词汇
